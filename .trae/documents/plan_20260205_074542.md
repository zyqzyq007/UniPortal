# Analysis and Fix for "Project Not Found" Errors

The user is reporting `404 Project not found` errors in two scenarios:
1.  **Deleting a project** from `ProjectsList.vue`.
2.  **Loading a project** in `ProjectHome.vue`.

This indicates a persistent issue where the frontend is trying to access projects using IDs that do not exist in the backend database.

## Root Cause Analysis
1.  **Mock Data vs. Real Data**: The application transitioned from using `mockStore` (localStorage) to a real backend API. However, the frontend might still be rendering old data if:
    -   The user is viewing a cached version of the page.
    -   The `ProjectsList` component rendered "mock" projects (e.g. `p-001`) that don't exist in the real database (which uses UUIDs).
    -   *Correction*: I see `getProjects` in `ProjectsList.vue` calls the API. If the API returns projects, they should be valid.
    -   **Critical Insight**: The user might have created projects *before* the backend was fully set up or connected, or the backend DB was reset (which happens with `dev.db` if not persisted properly or if I ran `prisma db push` which might reset/create new DB).
    -   If the frontend is hot-reloading (HMR), `projects` array might keep old state in memory? Unlikely on refresh.

2.  **ID Mismatch**:
    -   The user clicked a project. If the list was stale, it might have an old ID.
    -   If the user is on `http://localhost:3001/src/pages/projects/ProjectHome.vue...`, they might be clicking a direct link or refreshing a page with an old ID in the URL.

3.  **Backend Logic**:
    -   The backend logs (from previous turn) showed: `[DELETE] Project not found: <ID>`.
    -   This confirms the ID reaching the backend is indeed missing from the DB.

## Implementation Plan

### 1. Robust Error Handling in Frontend
-   **ProjectsList.vue**:
    -   If deletion fails with 404 (Project Not Found), we should **automatically remove it from the list**. It implies the project is already gone (maybe deleted by another user or DB reset).
    -   Show a message: "Project was already deleted or not found."

-   **ProjectHome.vue**:
    -   The current error handling (added in previous turn) shows a "Retry" or "Back" button. This is good.
    -   If the error is 404, we should explicitly guide the user to go back to the list, as "Retry" won't fix a missing ID.

### 2. Verify Data Consistency
-   The user should be advised to refresh the project list.
-   I will add a `force refresh` logic to `ProjectsList.vue` when it mounts to ensure no stale cache (though `onMounted` already fetches fresh).

### 3. Execution Steps
1.  **Update `ProjectsList.vue`**:
    -   Modify `handleDelete` to catch 404 specifically.
    -   If 404, remove the item from the local list and notify user "Project already deleted".
    
2.  **Update `ProjectHome.vue`**:
    -   Refine the error UI. If 404, hide "Retry" button (pointless) and highlight "Back to List".

3.  **Verify Backend**:
    -   The backend code looks correct. It checks ID and Owner.
    -   Ensure `dev.db` is stable.

## Detailed Changes
-   **File**: `src/pages/projects/ProjectsList.vue`
    -   Update `handleDelete` catch block.
-   **File**: `src/pages/projects/ProjectHome.vue`
    -   Update `error-actions` to conditionally show Retry.

This plan addresses the "why" (stale/missing IDs) by handling the "what now" (sync frontend state).
