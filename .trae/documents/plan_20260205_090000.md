## 现状分析
1. **项目文件结构树**：
   - 当前已使用 `FileTree.vue` 和 `TreeNode.vue`。
   - `TreeNode.vue` 中 `expanded` 默认值为 `true`，导致所有层级默认展开，不符合“默认展开第一级”的要求。
   - 连接线目前只有 `.children` 的 `border-left`，可能不够直观或缺少节点间的横向连线。
   - 响应式和动画已基本实现，但可能需要微调。

2. **项目名称显示**：
   - `DefaultLayout.vue` 中 `projectName` 仍在使用 `getProjectById` 从 `mockStore` 获取数据。由于项目已切换到真实后端，这里无法获取到项目信息（因为 `DefaultLayout` 没有发起 API 请求，且 `mockStore` 已不再维护或与真实数据不同步）。
   - 布局组件通常不直接请求数据，或者如果需要，应该有一个全局状态管理（如 Pinia 或简单的 `reactive` store）来存储“当前项目信息”。

## 改造方案

### 1. 完善文件树展示 (`TreeNode.vue`)
- **默认展开逻辑**：修改 `expanded` 的初始值。如果 `level === 1` 则默认为 `true`，否则为 `false`（或者受 `expandAll` 影响）。
- **连接线优化**：使用伪元素 (`::before`, `::after`) 在 CSS 中绘制更标准的树形连接线（垂直线 + 水平短线指向节点图标）。
- **样式调整**：确保缩进和线条在移动端也能正常显示。

### 2. 修复项目名称显示 (`DefaultLayout.vue`)
- **方案 A：全局状态**。创建一个简单的 `useProjectStore` (composable)，在 `ProjectHome` 加载项目成功后更新这个 store，`DefaultLayout` 响应这个 store 的变化。
- **方案 B：独立请求**。`DefaultLayout` 监听到 `route.params.projectId` 变化时，独立发起 `getProject` 请求。缺点是多一次请求，但解耦。
- **方案 C：Props/Emit**。由于 `RouterView` 的限制，传递比较麻烦。
- **选择方案 A**：最符合 Vue 3 组合式 API 的最佳实践。创建一个 `src/store/project.ts` (如果不引入 Pinia，就用 `reactive` 导出的单例)。

## 实施步骤
1.  **创建项目状态管理**：`src/store/projectState.ts`，包含 `currentProject` 和 `fetchProject` 方法。
2.  **更新 `ProjectHome.vue`**：在获取到项目数据后，更新全局状态。
3.  **更新 `DefaultLayout.vue`**：使用全局状态替代 mockStore，并监听路由变化以重置或重新获取（如果用户直接进入子路由，Layout 也应该能获取到名称，所以 Layout 自己监听 route 变化并尝试获取可能更稳健，或者由页面负责设置）。
    - *修正*：如果由 Layout 负责获取，那么所有子页面都不用操心 Title 的更新。Layout 监听 `projectId` 变化，如果有 ID，调用 API 获取详情并更新 Title。这样最稳妥。
4.  **优化 `TreeNode.vue`**：
    - 修改 `expanded` 默认值逻辑。
    - 添加 CSS 绘制树形连接线。
