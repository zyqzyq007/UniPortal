## 问题定位
用户指出通过“上传文件夹”方式上传项目后，概览页只显示文件而没有文件夹结构。这怀疑是上传逻辑中对文件夹路径处理不当导致的。

检查 [project.controller.ts](file:///Users/bytedance/kairos/tasks/UniPortal/server/src/controllers/project.controller.ts#L68-L78) 的上传逻辑：
```typescript
if (hasFolder) {
  for (const file of folderFiles) {
    const relativePath = file.originalname || file.filename; // <--- 关键点
    const targetPath = path.resolve(projectPath, relativePath);
    if (!targetPath.startsWith(projectPath)) { ... }
    fs.mkdirSync(path.dirname(targetPath), { recursive: true });
    fs.writeFileSync(targetPath, file.buffer);
  }
}
```
`multer` 上传时，`file.originalname` 默认只包含文件名（例如 `file.txt`），而不包含相对路径（例如 `subdir/file.txt`）。虽然前端使用了 `webkitdirectory`，但浏览器发送 FormData 时，如果不特殊处理，标准的 `file.name` 只是文件名。
**前端代码**（[ProjectsUpload.vue](file:///Users/bytedance/kairos/tasks/UniPortal/src/pages/projects/ProjectsUpload.vue#L226-L229)）已经正确处理了：
```typescript
const filePath = (file as any).webkitRelativePath || file.name
formData.append('folder', file, filePath) // 第三个参数是 filename
```
在 `FormData.append` 中传递第三个参数作为 `filename`，multer 接收到的 `file.originalname` 应该就是这个完整路径。

然而，如果 multer 配置或中间件没有正确保留这个路径，或者浏览器行为差异，可能导致后端收到的是扁平的文件名。
但更常见的问题是：**后端在保存文件时，是否真的创建了对应的目录结构？**
代码里有 `fs.mkdirSync(path.dirname(targetPath), { recursive: true });`，这说明如果 `relativePath` 包含路径分隔符，后端是会创建目录的。

**如果 `file.originalname` 只是文件名**：
那么所有文件都会被写到 `projectPath` 根目录下，导致目录结构丢失。这与用户描述的“只能解析出文件，而没有任何文件夹”完全吻合。

**验证怀疑**：
用户上传文件夹时，前端虽然传了 `webkitRelativePath` 到 `formData`，但后端 multer 处理后的 `file.originalname` 到底是什么？
通常 multer 会保留 `formData.append` 的第三个参数作为 `originalname`。

但还有一种情况：前端逻辑里 `selectedFolderFiles` 是 `File[]`。
在 [ProjectsUpload.vue](file:///Users/bytedance/kairos/tasks/UniPortal/src/pages/projects/ProjectsUpload.vue#L226-L229)：
```typescript
selectedFolderFiles.value.forEach((file) => {
  const filePath = (file as any).webkitRelativePath || file.name
  formData.append('folder', file, filePath)
})
```
这段看起来是正确的。

**那为什么后端收到的可能不对？**
可能是在某些浏览器或环境下，`webkitRelativePath` 为空，或者后端 Multer 配置问题。
或者，最关键的：**后端是否正确解析了包含目录的 originalname**。

让我们再看后端：
```typescript
const relativePath = file.originalname || file.filename;
```
如果前端传的是 `folder/sub/file.txt`，那么 `path.resolve(projectPath, relativePath)` 就会是 `.../storage/uuid/folder/sub/file.txt`，并且 `fs.mkdirSync` 会创建 `folder/sub`。

**如果用户反馈“没有文件夹”，那意味着 `relativePath` 里没有 `/`，所有文件都平铺了。**

**修复方案**：
我们需要确保前端发送的 `filename` 确实包含了路径，并且后端能正确接收。
如果前端已经发对了，那问题可能出在：用户上传时，**选择文件夹的根目录名通常不包含在 `webkitRelativePath` 中**（Chrome 行为：包含根目录名；Firefox：包含）。
等等，`webkitRelativePath` **包含** 选中的那个文件夹的名字作为前缀。
例如选了 `my-project` 文件夹，里面有 `src/index.js`，路径是 `my-project/src/index.js`。
后端保存时，会在 `storage/uuid/` 下创建 `my-project/src/index.js`。
这样结构是有的。

**除非**：前端获取 `webkitRelativePath` 失败。
检查前端代码：
```typescript
const filePath = (file as any).webkitRelativePath || file.name
```
如果 `webkitRelativePath` 缺失，就回退到 `file.name`（只有文件名），这就会导致扁平化！
**这正是问题所在。**

**如何解决？**
1.  **确认前端逻辑**：必须确保能拿到相对路径。对于文件夹上传，`webkitRelativePath` 是标准属性。
2.  **后端优化**：后端目前的逻辑是依赖 `originalname` 的。

**更深层的问题**：
如果用户上传的是 **压缩包**（Zip），解压后通常会保留目录结构。
如果用户上传的是 **文件夹**，依赖 `webkitRelativePath`。

**现在的现象是“上传文件夹”后没有目录结构。**
我需要确保前端在提交 `FormData` 时，`webkitRelativePath` 确实有值。

**另外一个可能性**：后端在处理 `multer` 数组时，可能因为某些配置丢失了文件名中的路径信息（虽然不太常见，但有些配置会以此为了安全）。
不过查看 [project.routes.ts](file:///Users/bytedance/kairos/tasks/UniPortal/server/src/routes/project.routes.ts)（之前看过的）里 multer 只是简单配置了 `memoryStorage`。

**最可疑的点**：
用户说“只能显示文件，而没有文件夹”。
如果是前端回退到了 `file.name`，那确实会这样。
但 `webkitdirectory` 选择的文件通常都有这个属性。

**修正思路**：
在前端上传逻辑中，显式打印一下 `filePath` 确认。
但作为修复，我们可以尝试**强制**要求文件夹上传必须有路径信息，或者在后端处理时，如果发现是文件夹上传（`folder` 字段），**信任**其路径结构。

**还有一个可能**：
后端在保存时，`path.resolve` 在某些系统（Windows vs POSIX）上对路径分隔符的处理差异。但现在环境是 macOS，应该都是 `/`。

**让我们仔细看用户的描述**：“目前只能解析出文件，而没有任何文件夹”。
这说明文件都在，但都在根目录下。
这 99% 是因为 `file.originalname` 丢失了路径前缀。

**修改计划**：
1.  **前端**：在 `ProjectsUpload.vue` 中，提交前再次确认 `webkitRelativePath` 的获取。
    实际上，`file.webkitRelativePath` 是只读属性。
    代码里 `formData.append('folder', file, filePath)` 是对的。
    **但是**，有些情况下（如拖拽上传，虽然这里没实现拖拽），或者特定浏览器，可能拿不到。
    但这里是 `<input type="file" webkitdirectory>`。

    **等等，还有一个问题**：
    如果用户选的是一个包含文件的文件夹，但文件都在该文件夹根目录下（例如 `dist/a.js`, `dist/b.css`）。
    `webkitRelativePath` 会是 `dist/a.js`。
    后端会创建 `dist` 目录。
    **如果用户说“没有文件夹”，那是连 `dist` 都没有吗？**
    如果连 `dist` 都没有，那说明 `originalname` 只是 `a.js`。

    **让我们检查 `project.controller.ts` 的 69-70 行**：
    ```typescript
    for (const file of folderFiles) {
      const relativePath = file.originalname || file.filename;
    ```
    这里直接用了 `originalname`。

    **我要做一个防御性修改**：
    在前端，显式地把路径作为单独的字段传给后端？不，multer 已经支持重写 filename。
    **或者**，我们可以改为上传一个 `filePaths` 的 JSON 字段，明确告诉后端每个文件的路径，而不依赖 `originalname` 的隐式传递。
    这样做最稳妥。

    **方案：显式传递路径映射**
    1.  前端：在 `formData` 中增加一个字段 `filePaths`，内容是 `JSON.stringify(pathsMap)`，或者直接按照顺序传递路径数组。
        或者更简单：`formData.append('paths', filePath)`，与 `folder` 文件数组一一对应。
    2.  后端：读取 `req.body.paths`。如果是数组，则与 `req.files.folder` 索引对应；如果是单个字符串（单文件），则对应一个。
        使用这个显式的路径来保存文件。

## 实施步骤
1.  **修改前端 `ProjectsUpload.vue`**：
    -   在遍历 `selectedFolderFiles` 时，不仅 append 文件，还 append 一个 `folderPaths` 字段（或者叫 `relativePaths`），值为 `filePath`。
2.  **修改后端 `project.controller.ts`**：
    -   在处理 `folderFiles` 时，尝试从 `req.body.folderPaths`（或类似字段）获取路径。
    -   如果存在显式路径，优先使用它；否则回退到 `originalname`。
    -   注意：`req.body` 在 `multer` 处理文件后才可用，且文本字段通常在文件字段之前或之后解析，取决于 multer 配置。但通常 `req.body` 会包含文本字段。
    -   *注意*：如果是同名字段多值，`req.body.folderPaths` 会是一个数组。需要确保顺序一致性。Multer 通常保持顺序。

    **更稳妥的方式**：
    前端构造 FormData 时，字段顺序很重要吗？不。
    但是 `folderPaths` 数组的顺序必须和 `folder` 文件的顺序一致。

    **改动**：
    **Frontend**:
    ```typescript
    selectedFolderFiles.value.forEach((file) => {
      const filePath = (file as any).webkitRelativePath || file.name
      formData.append('folder', file) // 不再依赖第三个参数 filename 的黑魔法
      formData.append('folderPaths', filePath) // 显式传递
    })
    ```
    **Backend**:
    ```typescript
    const folderPaths = req.body.folderPaths; // string | string[]
    // ...
    for (let i = 0; i < folderFiles.length; i++) {
      const file = folderFiles[i];
      let relativePath = file.originalname;
      if (folderPaths) {
         if (Array.isArray(folderPaths)) relativePath = folderPaths[i];
         else if (typeof folderPaths === 'string' && folderFiles.length === 1) relativePath = folderPaths;
      }
      // ...
    }
    ```

    这样能彻底解决路径丢失问题。
