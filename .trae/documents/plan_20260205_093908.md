## 问题定位
用户遇到的错误是 `MulterError: File too large`，这表明上传的文件（单个文件）超过了后端 `multer` 配置的大小限制。

检查 [project.routes.ts](file:///Users/bytedance/kairos/tasks/UniPortal/server/src/routes/project.routes.ts#L17-L19)：
```typescript
const upload = multer({
  storage: multer.memoryStorage(),
  limits: {
    fileSize: 50 * 1024 * 1024, // 50MB
  },
});
```
当前限制是 **50MB**。对于大型项目仓库（尤其是包含 `node_modules` 或大型资源文件时），这个限制很容易被突破。
此外，如果是上传文件夹（`folder` 字段），虽然单个文件可能不超过 50MB，但 `memoryStorage` 可能会导致 Node.js 进程内存溢出（如果并发上传大量文件）。
但这里的报错是明确的 `File too large`，说明是单文件限制。

## 修复方案
1.  **增大单文件大小限制**：将 `fileSize` 限制提高，例如到 **500MB** 或 **1GB**。
2.  **增大请求体总大小限制**：Express 默认也有 body 大小限制，虽然 multer 处理 multipart 时主要受自身 limit 控制，但如果使用了 `body-parser` 或其他中间件也需注意（multer 通常绕过 body-parser）。
3.  **（可选）优化存储方式**：目前使用 `memoryStorage`（内存存储），对于大文件上传非常危险，极易导致 OOM (Out Of Memory)。建议改为 `diskStorage`（磁盘存储）或流式处理。鉴于用户目前遇到的是 Size 限制，优先解决 Size 问题。如果用户上传超大项目，改为 `diskStorage` 是更稳健的长久之计。

**本次行动**：
1.  将 `fileSize` 限制提升至 **500MB**（满足大多数源码文件需求）。
2.  将存储引擎从 `memoryStorage` 切换为 `diskStorage`，直接写入临时目录，避免内存崩溃。
    -   *注意*：如果改为 `diskStorage`，`req.files` 中的对象将包含 `path` 而不是 `buffer`。
    -   我们需要同步修改 [project.controller.ts](file:///Users/bytedance/kairos/tasks/UniPortal/server/src/controllers/project.controller.ts)，从 `file.path` 读取文件或移动文件，而不是读取 `file.buffer`。

**决策**：
为了快速修复用户的 `File too large` 报错，最直接的是改 `fileSize`。
但为了系统稳定性（避免 500MB 文件撑爆内存），**必须**同步切换到 `diskStorage`。

    -   *注意*：如果是上传文件夹，multer 会把所有文件先存到 temp。我们处理时再移过去。

这将是一个涉及后端核心上传逻辑的重构，但非常有必要。

3.  **稳妥方案**：改为 `diskStorage`。

**文件修改**：
1.  `server/src/routes/project.routes.ts`: 修改 multer 配置。
2.  `server/src/controllers/project.controller.ts`: 修改文件读取逻辑 (`file.buffer` -> `fs.readFileSync(file.path)` 或流式移动)。

**代码细节**：
Controller 中目前的逻辑：
```typescript

```
改为：
```typescript

```
这样既解决了内存问题，也解决了大小限制问题。
