## 问题确认
用户反馈“没有打开的文件夹的icon `svg`，现在仍然显示成了打开状态”。
我刚刚将箭头图标从字符 `▶` 换成了 SVG `chevron-right.svg`。
CSS 逻辑是：
```css
.arrow {
  transform: rotate(0deg); /* 默认 0 度 */
}
.arrow.expanded {
  transform: rotate(90deg); /* 展开 90 度 */
}
```
**如果用户看到“未打开的文件夹”显示成了“打开状态”（即向下指的箭头）**，可能有以下原因：
1.  **SVG 图标本身的方向问题**：我创建的 `chevron-right.svg` 实际上画的是向下指的箭头吗？
    检查刚才写入的 SVG 内容：
    ```xml
    <path ... d="M6.22 3.22a.75.75 0 0 1 1.06 0l4.25 4.25..."/>
    ```
    这是一个向下的箭头（通常 chevron-down 的路径数据）。
    **糟糕**，我可能用了一个向下的路径数据命名为了 `chevron-right.svg`，或者反之。
    通常 `chevron-right` 是指向右侧的。如果我用了向下的路径，那么 `rotate(0deg)` 时它就是向下的（打开状态），而 `rotate(90deg)` 会指向左边。

    让我们验证一下路径数据：
    `M6.22 3.22 ... l4.25 4.25 ... l-4.25 4.25`
    这看起来像是：从 (6,3) 到 (10,7) 再到 (6,11)。这是一个指向 **右边** 的箭头（顶点在右侧）。
    X 坐标变大，Y 坐标变大再变小？不对，SVG 坐标系 Y 轴向下。
    (6,3) -> (10,7) -> (6,11)
    这是一个 `>` 形状。所以路径是对的（指向右）。

2.  **默认状态问题**：
    如果路径是对的（向右），那么 `rotate(0deg)` 应该是向右（闭合）。
    如果用户看到它是“打开状态”（向下），说明：
    -   要么 `expanded` 为 `true`。
    -   要么 CSS 旋转逻辑反了。
    -   要么用户认为 `>` 是打开状态？不，用户明确说“显示成了打开状态”。

    检查 `TreeNode.vue`：
    ```typescript
    const expanded = ref(!!props.expandAll)
    ```
    默认是 `false`（如果 `expandAll` 未传）。

    **再检查 SVG**：
    `M6.22 3.22 ...`
    如果是向右的，那是对的。
    **但是**，如果我之前的字符是 `▶`，它是实心的。现在的 SVG 是线性的 `>`。
    用户说“显示成了打开状态”，在 VS Code 里：
    -   折叠：`>` (chevron-right)
    -   展开：`v` (chevron-down)

    **可能的原因**：
    用户看到的“打开状态”是指**文件夹图标**（FolderOpen）？
    用户说的是“文件夹的icon `svg`”。
    代码：
    ```javascript
    if (node.type === 'dir') {
      return expanded.value ? FolderOpenIcon : FolderIcon
    }
    ```
    如果 `expanded` 是 false，应该是 `FolderIcon`（闭合文件夹）。

    **等等，用户选中的元素是**：
    ```html
    <svg ... class="arrow expanded">...</svg>
    ```
    **关键点**：`class="arrow expanded"`。
    这说明 `expanded` 确实是 `true`！
    这意味着在渲染时，`expanded` 被设置为了 `true`。

    **为什么？**
    1.  `props.expandAll` 是 true？
    2.  `props.level` 逻辑？之前的逻辑是 `level === 1` 默认 true，但我改成了 `!!props.expandAll`。
    3.  **热更新残留**？Vue 组件的 `ref` 在 HMR 时可能会保留状态。
    4.  **父组件传参**？`FileTree.vue` 里是否传了 `expandAll`？
        检查 `FileTree.vue`。

    如果 `FileTree.vue` 传了 `expandAll`，或者在递归组件 `TreeNode.vue` 中传递了 `expandAll`。
    检查 `TreeNode.vue` 的递归调用：
    ```html
    <TreeNode ... :expandAll="expandAll" ... />
    ```
    它透传了 `expandAll`。

    **检查 `FileTree.vue`**：
    我看不到 `FileTree.vue` 的最新内容，但我之前加了“全部展开/收起”按钮。
    如果那个按钮对应的状态是 `true`，那么所有节点都会展开。

    **但是**，用户是刚刷新或刚打开页面。
    如果 `FileTree` 里的 `expandAll` 默认是 `false`，那应该没问题。

    **还有一个可能**：
    我刚才的修改：
    ```typescript
    const expanded = ref(!!props.expandAll)
    ```
    如果 `props.expandAll` 是 `undefined`，`!!undefined` 是 `false`。

    **让我们看看用户的截图中 `outer_html`**：
    `<svg ... class="arrow expanded">`
    这确凿无疑地表明 `expanded` 类存在。

    **更诡异的可能性**：
    CSS 问题。
    ```css
    .arrow { transform: rotate(0deg); }
    .arrow.expanded { transform: rotate(90deg); }
    ```
    如果 `class` 有 `expanded`，它就转了 90 度。

    **结论**：组件的状态 `expanded` 确实是 `true`。
    **为什么是 true？**
    可能是因为我在 `FileTree.vue` 或其他地方没有正确重置，或者之前的 HMR 导致状态没清。
    但用户说“修改成未打开的状态”，意味着他希望**默认**是关的。
    我已经改了 `const expanded = ref(!!props.expandAll)`。

    **等等，我可能没有刷新页面？**
    不，用户是在浏览器里选中的。
    
    **有没有可能**：我之前的修改 `props.level === 1` 虽然删了，但 `TreeNode` 内部逻辑有缓存？
    或者，**FileTree** 传递了 `expandAll`？

    **检查 `FileTree.vue`**
    我需要读取 `src/components/FileTree.vue` 看看它怎么使用 `TreeNode` 的。

    另外，用户说“修改成未打开的状态”，可能是指**SVG 图标本身**看起来像打开的？
    如果我的 SVG 画的是向下箭头（`v`），那么：
    -   默认（0度）：向下（打开状）。
    -   expanded（90度）：向左。
    
    如果我画的是向右箭头（`>`），那么：
    -   默认（0度）：向右（折叠状）。
    -   expanded（90度）：向下（打开状）。

    我刚才确认了 SVG 路径看起来是向右的。
    但如果用户说“没有打开...显示成了打开状态”，那可能是我**看错了 SVG 路径**，或者旋转逻辑反了。

    **再次检查 SVG**
    `M6.22 3.22 ... L9.94 8 ...`
    (6.22, 3.22) -> (10, 8) approx.
    这是一个向下的斜线。
    (9.94, 8) -> (6.22, 12.something)
    这是一个向下的斜线（往回）。
    所以是 `>` 形状。
    (6,3) --\
             (10,8)
    (6,12) --/
    
    是的，是向右的。

    **那么问题只能是 `expanded` class 被加上了。**
    用户提供的 HTML 片段里确实有 `class="arrow expanded"`。
    这意味着 Vue 实例的 `expanded` 值为 `true`。

    **根本原因猜测**：
    1.  `FileTree` 组件中 `expandAll` 默认为 `true`？
    2.  或者 HMR 没生效？
    3.  或者我在 `TreeNode` 里的逻辑有漏洞？
