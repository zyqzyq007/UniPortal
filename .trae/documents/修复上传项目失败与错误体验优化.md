## 现状与原因分析
- 前端上传页 [ProjectsUpload.vue](file:///Users/bytedance/kairos/tasks/UniPortal/src/pages/projects/ProjectsUpload.vue) 目前只调用 `createProject` 并发送 JSON（name/description/repoUrl），没有真正上传文件或文件夹。
- 后端 `POST /api/projects` 在 [project.controller.ts](file:///Users/bytedance/kairos/tasks/UniPortal/server/src/controllers/project.controller.ts) 仅支持 JSON，并且强依赖 `req.user`（鉴权中间件）。如果用户未登录或 token 不在请求里，前端会收到失败；另外文件上传也未被处理。
- 截图显示 “Internal server error”，说明后端在处理请求时抛错，但前端只能展示笼统错误，无法引导用户修复。

## 计划与改造范围
### 1. 后端：完善上传接口与错误返回
- 为 `POST /api/projects` 增加对 multipart/form-data 的支持（例如使用 `multer`）。
- 允许三种互斥上传方式：
  1) repoUrl
  2) zip/tar 文件
  3) 文件夹（多文件上传）
- 校验逻辑：若三者不满足“仅一种”，返回 400，包含明确错误信息。
- 添加清晰错误响应结构：`{ code, message, detail }`，避免模糊 500。

### 2. 前端：修复提交失败与增强 UX
- 使用 `FormData` 发送上传请求，确保文件/文件夹真实上传。
- 提交时增加加载状态（按钮禁用 + 文案变化）。
- 增加失败状态 UI（红色提示框 + 图标），错误信息从后端 detail 显示。
- 保留用户已填写内容（当前已保留），增加“重试”和“取消”按钮。
- 区分错误类型（网络、鉴权、验证、服务器），并给出更明确提示。

### 3. 组件适配
- 更新上传组件逻辑，支持：
  - repoUrl 上传
  - zip/tar 上传
  - 文件夹上传
- 上传成功后返回项目列表并提示成功状态。

### 4. 测试与验证
- 模拟失败场景（无登录、缺少名称、多种方式同时选择、网络断开、服务器错误）。
- 验证删除最后一项时 UI 的正常刷新。
- 前后端联调，确保响应码与错误提示一致。

## 实施步骤
1. 后端新增文件上传解析与校验逻辑。
2. 前端改为 FormData + 错误分类显示。
3. UI 增加错误提示与重试/取消按钮。
4. 完整联调测试。
